WEEK 03

CLASS 02 · Reliability Engineering

Topics: Fault → Error → Failure, Metrics (POFOD, ROCOF)

My Evidence (V2):

System: Emergency Medical Dispatch Center (997)
System Profile
  • Users: Dispatch operators and emergency paramedics.
  • Usage Pattern: High-stakes, unpredictable bursts of critical communications.
Chosen Metric: POFOD (Probability of Failure on Demand)
  • Justification: For an emergency dispatch system, it does not matter how long the system runs without issues (ROCOF); what matters is that it must work the exact moment an emergency call is received. A failure upon demand translates directly to a loss of life.
Proposed Strategy: N-Version Programming
  • To ensure the dispatch routing logic never fails, the core routing module will be built using N-Version programming. Multiple independent algorithms will calculate the nearest ambulance, and a voting system will ensure the correct route is dispatched even if one software version encounters a fault.

WEEK 05

CLASS 04 · Security Engineering

Topics: CIA Triad, Security Assurance Case, Misuse Cases

My Evidence (V2): National Citizen Identity Portal (Nafath-style)

1. System Description & Threat Analysis
  • Asset: Centralized citizen identification records.
  • Threat Vector: Session hijacking and phishing attacks attempting to bypass authentication.
  • Misuse Case: An attacker intercepts a valid user token on a public network to impersonate a citizen and authorize fraudulent government services.
2. Design Rationale (Zero-Trust Architecture)
  • Vulnerability Avoidance: Enforcing strict Time-to-Live (TTL) limits on authentication tokens (e.g., 5-minute expiry).
  • Compartmentalization: Implementing biometric multi-factor authentication (MFA) for every new service request, ensuring a stolen password alone is useless.
3. Verification & Validation
  • Penetration Testing: Simulating Man-in-the-Middle (MITM) attacks to verify that TLS 1.3 encryption prevents token extraction.
  • Audit Trails: Automated logging of all login attempts to detect and block brute-force patterns instantly.
WEEK 06

CLASS 05 · Resilience Engineering

Topics: Recognition & Resistance, Decoupling, Sociotechnical Factors

My Evidence (V2): Cloud-Based Airline Booking Engine

1. Failure Chain Analysis
  • Fault: A memory leak in the external payment gateway API.
  • Error: Booking transactions hang, tying up server resources during a flash sale.
  • Failure: The entire reservation database locks up, preventing all customers and ticketing agents from booking flights.
2. Recognition & Resistance (R&R) Strategy
  • Recognition: Implementation of Circuit Breakers that monitor API response times. If times exceed 2 seconds, the circuit trips.
  • Resistance: The system stops accepting new payment requests temporarily (Fail-Soft), presenting users with a "Queue" screen while keeping the rest of the site (flight browsing) fully operational.
3. Human-in-the-Loop Override
  • If the automated system fails at the airport gate, ticketing agents can switch to a local cached manifest to manually board passengers offline, ensuring operational continuity despite cloud outages.
WEEK 07

CLASS 06 · Software Reuse

Topics: SPL, The Ariane 5 Lesson

My Evidence (V2): Smart City Fleet Management (SPL)

Software Product Line (SPL) Approach
  • Core Components: GPS tracking, user authentication, and data logging are reused across all city departments.
  • Specialized Components: Custom modules are swapped depending on the fleet (e.g., siren control for police, route optimization for waste management).
  • Justification: SPL drastically reduces development time and ensures all municipal vehicles share a standardized, highly tested security architecture.
Validation & The Ariane 5 Lesson
  • The Risk: Reusing a legacy GPS tracking module designed for slow garbage trucks inside high-speed police interceptors could cause data buffer overflows (similar to the Ariane 5 disaster).
  • The Solution: Strict boundary fuzzing and invariant testing to ensure reused code can handle the new physical environment's speed and data rate requirements before deployment.

WEEK 09

CLASS 08 · Distributed Software Engineering

Topics: Client-Server Architecture, Failover, Security

My Evidence (V2): National E-Health Record System

Distributed Architecture Design
  • Client Nodes: Hospital workstation terminals processing UI and local validation.
  • App Server Nodes: Regional cloud instances handling business logic.
  • Data Store Nodes: Centralized databases with regional shards for speed.
Failover & Security
  • Stateless Microservices: App nodes hold no local state. If a regional server crashes, a load balancer instantly reroutes traffic to a healthy node.
  • Data-in-Transit: Enforcing strict mTLS (Mutual TLS) to authenticate node-to-node communication, ensuring patient data cannot be intercepted across the wide area network.
WEEK 10

CLASS 09 · System of Systems & Multi-tier Architecture

Topics: Middleware, Failure Isolation

My Evidence (V2): E-Commerce Logistics Platform

Multi-Tier Implementation
  • Presentation Tier: Mobile app for delivery drivers.
  • Application Tier: Cloud servers running intensive route-matching algorithms. Offloading this from the mobile device saves battery and standardizes performance.
  • Data Tier: Inventory and user databases.
The Role of Middleware
  • Utilizing message queues (like Kafka or RabbitMQ) between the App and Data tiers ensures asynchronous processing. If the database experiences a spike, orders are queued safely in the middleware rather than crashing the driver's app, ensuring complete failure isolation.

🧠 Reflection

❓ What changed in how I think?

I transitioned from viewing software as just "writing code" to understanding it as "building architectural proof." Engineering is about anticipating failure and proving, through evidence and documentation, that your system can handle it safely.

⚖️ Safety vs Reliability

The realization that a system can function exactly as programmed (Reliable) but still cause catastrophic harm (Unsafe) fundamentally changed my approach. Requirements must be challenged for safety, not just implemented blindly.

💼 Professional Carry-over

I will carry the principle of the "Safety Case" into my professional career. Whether I'm building IT infrastructure or working in aviation tech, I now understand that the burden of proving a system is dependable rests entirely on the engineer.
